# GCE(Google Compute Engine)
## Compute Engine
仮想マシンインスタンスを提供するIaaSのサービス。
LinuxやWindowsなどのOSに対応し、CPU/メモリ/ストレージを自由に構成することができる。
また、GCPのグローバルなネットワークインフラ上で起動させることで、高可用性設計が可能。

### インスタンスのタイプ
インスタンスを作成する際には、利用目的に応じてマシンシリーズやマシンタイプを指定する。それぞれの用語の特徴は以下。

|EC2用語|概要|GCE用語（新体系）|例|
|:----|:----|:----|:----|
|インスタンスファミリー|VMの用途・性能バランスごとの大分類|マシンファミリー（H）|H（Hyperdisk対応）|
|世代/シリーズ|同じファミリー内で世代や構成の違いを表す|マシンシリーズ（H3）|H3（第3世代）|
|インスタンスタイプ|vCPUとメモリ量で個別に識別される構成名|マシンタイプ（standard-88）|h3-standard-88|


### ストレージオプション
VMを作成する際OSなどが入っているブートディスクはデフォルトで作成される。
ブートディスクは別に外部ディスクをマウントすることができ、これを`ストレージオプション`と呼ぶ。

大きく3つのストレージオプションが準備されている。
また、それぞれについてもバランスタイプやエクストリームタイプなど種別が存在。

|種類|特徴|向いている用途|
|:----|:----|:----|
|Persistent Disk|標準的な高可用ディスク|汎用データ保存（アプリ、ログ、DB等）|
|Hyperdisk|高スループット・IOPS可変|データベース、分析ワークロード|
|Local SSD|超高速・揮発性（VM停止で消える）|キャッシュ、テンポラリ処理|

#### Regional Persistent Disk
IOPSのスペックが容量に応じて変化する
- 30 IOPS / GB（目安）
- 最大 100,000 IOPS
    - 80GB → 約 2,400 IOPS
    - 500GB → 約 15,000 IOPS


HAのためのディスクで以下の特徴がある
- 同一リージョン内の2ゾーンに同期レプリケーション
- 常に最新データが2箇所に存在
- 片方のゾーンが死んでももう一方が即利用可能



### バックアップ
インスタンス（VMおよび外部ストレージ）のバックアップについては`スナップショット`、`マシンイメージ`の方法がある。
- スナップショット：ストレージのバックアップ
- マシンイメージ：環境全体保存でインフラ構成ごと再現

| 項目 | スナップショット | マシンイメージ |
|------|------------------|----------------|
| 保存単位 | ディスク単体 | VM全体 |
| 保存対象 | Persistent Diskの中身 | ディスク + VM構成情報 |
| マシンタイプ | 保存されない | 保存される |
| ネットワーク設定 | 保存されない | 保存される |
| サービスアカウント | 保存されない | 保存される |
| メタデータ | 保存されない | 保存される |
| 追加ディスク | 個別に取得 | まとめて保存可能 |
| バックアップ方式 | 増分 | フル構成保存 |
| 主目的 | データ保護 | VM完全復元 |
| DR用途 | △（再構築必要） | ◎（ほぼワンクリック復旧） |
| コスト | 比較的安価 | 構成情報含むためやや高め |
| 復旧手順 | VMを再作成しディスク復元 | ほぼそのまま再作成 |

### MIG(Managed Instance Group)
Managed Instance Group（MIG）とは、同一構成の Compute Engine インスタンスをテンプレートから自動生成・管理する仕組み

MIGの本質は次の3点：
- インスタンステンプレートに基づく宣言的管理
- 自動修復（Autohealing）
- オートスケーリング対応

つまりMIGは、「同じ構成のVM群を“あるべき状態”に保ち続けるコントローラ」

MIGを作成する際の流れは以下
```
カスタムイメージ  
↓  
インスタンステンプレート  
↓  
MIG  
↓  
VM群
```

具体的にMIGを設定する際の流れは以下の表の通り。

| ステップ | 何をするか | 目的 | なぜ必要か |
|-----------|------------|------|------------|
| ① | VMを構築 | ベース環境作成 | アプリや設定を作り込む |
| ② | カスタムイメージ作成 | ゴールデン化 | 同一OS環境を再利用可能にする |
| ③ | インスタンステンプレート作成 | VM設計図定義 | マシンタイプ・ネットワークを固定 |
| ④ | MIG作成 | VM群生成 | 宣言的管理を開始 |
| ⑤ | オートスケール設定 | 自動増減 | ピーク対応 |
| ⑥ | ヘルスチェック設定 | 自動修復 | 可用性確保 |

インスタンステンプレートでは、カスタムイメージを指定するためカスタムイメージを作成後、インスタンステンプレートを作成する必要がある。

| 構成要素 | 内容 |
|------------|------|
| マシンタイプ | e2-mediumなど |
| ネットワーク | VPC・サブネット |
| サービスアカウント | IAM権限 |
| ディスク設定 | ブートディスクサイズなど |
| 参照イメージ | カスタムイメージ or 公開イメージ |


### 費用
費用に関しては重要課金制である。1分以降は秒単位での課金が発生する。
費用の割引としては以下を意識しておくと良い。
- 継続利用割引・・・1ヶ月で25%以上利用すると自動で割引される
- 確約利用割引・・・1年/3年で利用を確約することで割引を受けることができる

### Compute Engineの高可用性
GCEでは障害発生時の対応として大きく2種類の対応が存在。
`ライブマイグレーション`はユーザー側として意識することはないが、`ホストエラー`については、ハードウェア障害としてアプリに影響するため高可用性を担保するためには単一障害点とならない構成にする必要がある。

|項目|ライブマイグレーション|ホストエラー|
|:----|:----|:----|
|内容|物理ホストのメンテナンス前にVMを事前に別ホストへ移動|物理ホストが突然ダウンしてVMが強制停止|
|トリガー|Googleによる予防的なメンテナンス|ハードウェア障害などによる予期せぬ停止|
|VMの状態|停止せずに稼働し続ける（ほぼ無停止）|一時的に停止またはクラッシュ（再起動が必要）|
|GCPの対応|自動で別ホストにVMを移行|自動的に再起動される（オプション設定による）|


### BYOL(Bring Yout Own Licence)
#### Windows Server
WindowsServerのオンプレミスライセンスは、共有基盤（マルチテナント）では利用できない。
占有物理ホスト（Dedicated/Sole Tenancy）が必要になる。

以下コマンドを利用して、インポートすることで、このディスクはBYOL用Windowsであることを指定する。
```
--os=windows-2022-dc-v
```


### OS Config
OS Configは、Compute Engine VMのOSを
- 可視化
- 統制
- 自動更新

するための管理プレーン機能。

GoogleがOSそのものを管理するのではなく、OSの標準更新機構を活用しながら運用を自動化する仕組みを提供。

#### Patch Management（パッチ管理）
OS Configにおける最も重要な機能で、以下を実行することができる。
- 利用可能なアップデートの検出
- パッチの一括適用
- スケジュール実行
- 再起動ポリシー制御
- 適用結果の可視化

Debianであれば apt、RHELであれば yum など、OSネイティブの仕組みを使って更新が行われます。
つまり、SSHしてapt updateを打つ作業を、管理プレーンから制御できる機能。



### アップデートポリシー
MIGのアップデートは2つの更新モードがある
- Proactive: 既存インスタンスの順次置き換え
- Opportunistic: スケールアウトや際作成時に適用



## Migrate to Virtual Machines概要

オンプレミスの VMware 環境から Google Cloud の Compute Engine へ  
リフト＆シフト移行を行う際、Google が公式に推奨しているのが  
Migrate to Virtual Machines。

このツールの本質は、「VM単体のコピー」ではなく「業務システム単位で安全に移行すること」 にある。

Migrate for Compute Engine では、移行を以下の4段階で整理している。

| フェーズ | 目的 | 具体的にやること | ポイント |
|---|---|---|---|
| アセスメント | 現状把握 | VM一覧、リソース使用量、通信関係、構成の把握 | VM単位ではなくシステム単位で捉える |
| 移行計画 | 設計 | 移行対象のグルーピング、順序、テスト／本番の切り分け | 起動順序・依存関係を明確化 |
| Runbook作成 | 手順の定義 | 移行手順をシナリオとして定義 | 手作業を排除し再現性を確保 |
| 自動移行 | 実行 | Runbookに基づく移行実行 | 移行が「作業」ではなく「実行」になる |


### 移行計画と依存関係の考え方

業務システムでは VM は単独で動作しない。

例：
- DB層
- AP層
- Web層

のように依存関係を持つため、
- DB → AP → Web の順で移行
- 同一層はまとめて移行

といった 順序とまとまり を考慮する必要がある。

Migrate for Compute Engine は、この依存関係を考慮した移行を前提としている。


### Runbook ベース移行とは何か
Runbook とは、移行作業を定義した 実行シナリオ である。
- どの VM グループを
- どの順番で
- テスト移行か本番移行か

を事前に定義し、同じ手順を何度でも安全に実行可能 にする。
