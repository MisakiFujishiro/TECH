7. GCE(Google Compute Engine)¶

7.1. Compute Engine¶

仮想マシンインスタンスを提供するIaaSのサービス。 LinuxやWindowsなどのOSに対応し、CPU/メモリ/ストレージを自由に構成することができる。 また、GCPのグローバルなネットワークインフラ上で起動させることで、高可用性設計が可能。

7.1.1. インスタンスのタイプ¶

インスタンスを作成する際には、利用目的に応じてマシンシリーズやマシンタイプを指定する。それぞれの用語の特徴は以下。

7.1.2. ストレージオプション¶

VMを作成する際OSなどが入っているブートディスクはデフォルトで作成される。 ブートディスクは別に外部ディスクをマウントすることができ、これをストレージオプションと呼ぶ。

大きく3つのストレージオプションが準備されている。 また、それぞれについてもバランスタイプやエクストリームタイプなど種別が存在。

7.1.2.1. Regional Persistent Disk¶

IOPSのスペックが容量に応じて変化する

  • 30 IOPS / GB（目安）

  • 最大 100,000 IOPS

    • 80GB → 約 2,400 IOPS

    • 500GB → 約 15,000 IOPS

HAのためのディスクで以下の特徴がある

  • 同一リージョン内の2ゾーンに同期レプリケーション

  • 常に最新データが2箇所に存在

  • 片方のゾーンが死んでももう一方が即利用可能

7.1.3. Firewall¶

GCE の Firewall は、 VPC ネットワーク単位で適用される、ステートフルなパケットフィルタ

重要ポイント：

  • VM単位ではなく VPC単位

  • ルールは インスタンスに紐づく（タグ / サービスアカウントで対象指定）

  • ステートフル（stateful）

    • 片方向が許可されると、戻りトラフィックは自動許可

評価順序は以下：

  • 優先度（小さい数値が優先）

  • 条件一致

  • 最初に一致したルールで決定（それ以降は評価されない）

また、Firewallには以下のような暗黙的なルールが設定されている

7.1.5. MIG(Managed Instance Group)¶

Managed Instance Group（MIG）とは、同一構成の Compute Engine インスタンスをテンプレートから自動生成・管理する仕組み

MIGの本質は次の3点：

  • インスタンステンプレートに基づく宣言的管理

  • 自動修復（Autohealing）

  • オートスケーリング対応

つまりMIGは、「同じ構成のVM群を“あるべき状態”に保ち続けるコントローラ」

7.1.6. 費用¶

費用に関しては重要課金制である。1分以降は秒単位での課金が発生する。 費用の割引としては以下を意識しておくと良い。

  • 継続利用割引・・・1ヶ月で25%以上利用すると自動で割引される

  • 確約利用割引・・・1年/3年で利用を確約することで割引を受けることができる

7.1.7. Compute Engineの高可用性¶

GCEでは障害発生時の対応として大きく2種類の対応が存在。 ライブマイグレーションはユーザー側として意識することはないが、ホストエラーについては、ハードウェア障害としてアプリに影響するため高可用性を担保するためには単一障害点とならない構成にする必要がある。

ホストエラー

7.1.8. BYOL(Bring Yout Own Licence)¶

7.1.8.1. Windows Server¶

WindowsServerのオンプレミスライセンスは、共有基盤（マルチテナント）では利用できない。 占有物理ホスト（Dedicated/Sole Tenancy）が必要になる。

以下コマンドを利用して、インポートすることで、このディスクはBYOL用Windowsであることを指定する。

--os=windows-2022-dc-v

7.1.9. OS Config¶

OS Configは、Compute Engine VMのOSを

  • 可視化

  • 統制

  • 自動更新

するための管理プレーン機能。

GoogleがOSそのものを管理するのではなく、OSの標準更新機構を活用しながら運用を自動化する仕組みを提供。

7.1.9.1. Patch Management（パッチ管理）¶

OS Configにおける最も重要な機能で、以下を実行することができる。

  • 利用可能なアップデートの検出

  • パッチの一括適用

  • スケジュール実行

  • 再起動ポリシー制御

  • 適用結果の可視化

Debianであれば apt、RHELであれば yum など、OSネイティブの仕組みを使って更新が行われます。 つまり、SSHしてapt updateを打つ作業を、管理プレーンから制御できる機能。

7.1.10. アップデートポリシー¶

MIGのアップデートは2つの更新モードがある

  • Proactive: 既存インスタンスの順次置き換え

  • Opportunistic: スケールアウトや際作成時に適用

7.2. Migrate to Virtual Machines概要¶

オンプレミスの VMware 環境から Google Cloud の Compute Engine へ
リフト＆シフト移行を行う際、Google が公式に推奨しているのが
Migrate to Virtual Machines。

このツールの本質は、「VM単体のコピー」ではなく「業務システム単位で安全に移行すること」 にある。

Migrate for Compute Engine では、移行を以下の4段階で整理している。

フェーズ

目的

具体的にやること

ポイント

アセスメント

現状把握

VM一覧、リソース使用量、通信関係、構成の把握

VM単位ではなくシステム単位で捉える

移行計画

設計

7.2.1. 移行計画と依存関係の考え方¶

業務システムでは VM は単独で動作しない。

例：

  • DB層

  • AP層

  • Web層

のように依存関係を持つため、

  • DB → AP → Web の順で移行

  • 同一層はまとめて移行

といった 順序とまとまり を考慮する必要がある。

Migrate for Compute Engine は、この依存関係を考慮した移行を前提としている。

7.2.2. Runbook ベース移行とは何か¶

Runbook とは、移行作業を定義した 実行シナリオ である。

  • どの VM グループを

  • どの順番で

  • テスト移行か本番移行か

を事前に定義し、同じ手順を何度でも安全に実行可能 にする。
