# SoftWareArchitecture
ソフトウェアアーキテクチャー(SWA)について明確な定義はない。
SWAとは、システムの構造、アーキテクチャ特性（満たすべき非機能要件）、ルールやガイドをまとめてSWAと呼ばれる。
SWAの範囲は非常に広く、全てを把握することは難しいことを踏まえると、幅広い知識をまずは知っておくことが必要（フレームワーク・言語・設計パターン・CICD・開発手法）

SWAを考えないと、踏襲や会社標準などにより、システムがパフォーマンスを満たせなかったり、コストがかかるなどの課題が発生する。

## アーキテクチャパターン
SWAのベストプラクティスを再利用するための、アプリケーションやシステムの構築パターン
- マイクロサービスアーキテクチャ
- 三層アーキテクチャ

## デザインパターン
SW開発の実装におけるノウハウの設計や実装パターン
- Singletonパターン
- Mediatorパターン

## フレームワーク
一般的な機能や提携コードをライブラリとして準備したソフトウェアの骨組み
- Spring
- React

# マイクロサービスアーキテクチャ
マイクロサービスの大前提
- 疎結合
    - サービス間は「約束されたインターフェース」を通じて通信する
    - 実装の詳細は外部に漏らさない
    - マイクロサービスの実装と呼び出し元の間に API 契約（API contract）を作成する
- 独立した進化を前提にする
    - 各サービスは独立して変更される
    - 他サービスのリリースを待たない
    - バージョン管理スキームを実装する
- 技術や言語に縛られない
    - 各サービスは最適な技術を選べる
    - 同じ言語である必要はない
